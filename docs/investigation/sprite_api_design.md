# スプライトAPI設計検討: Image拡張 vs Sprite新設

エンジンの高レベル描画APIとして `sprite()` を導入し、切り出し領域(`u, v, w, h`)や色変換情報をオブジェクト側に持たせるにあたり、「既存の `Image` クラスを拡張する」か「新しく `Sprite` クラスを用意する」かのPros & Consを比較します。

## Option A: 既存の `Image` クラスを拡張する
`Image` クラス内に `u, v, w, h` (デフォルトは画像全体) や `tint` などのプロパティを持たせる手法。

### Pros (メリット)
* **APIがシンプルになる**: クラスが1つで済むため、利用者が覚える概念が少なくて済みます。
* **既存コードとの互換性**: 現在 `Image.load()` で読み込んでいる変数をそのまま `sprite()` 描画に渡しやすくなります。
* **C拡張実装が単純**: 扱うオブジェクトが1種類なので、MicroPythonのCモジュール化の際の実装が少しシンプルになります。

### Cons (デメリット)
* **メモリ効率とバッファ共有の課題**: 1つのスプライトシート（例: `faces.png`）から複数の異なる顔スプライトをオブジェクトとして定義したい場合、それぞれが `Image` インスタンスになります。このときピクセルバッファをディープコピーしてしまうとRAMを大量に消費するため、「バッファを浅いコピーで共有する仕組み」を独自に実装する必要があります。
* **役割の混同**: 「テクスチャデータの実体（ピクセルバッファ）」と「その一部を切り出すビュー（View）」という2つの役割が一つのクラスに混在し、単一責任原則(SRP)からやや外れます。

---

## Option B: 新たに `Sprite` (または `SubImage`) クラスを新設する
`Image` クラスは純粋にメモリ上のピクセルバッファのみを管理し、新しい `Sprite` クラスが「`Image`への参照」「切り出し領域 `u, v, w, h`」「色変換情報」を保持する手法。

### Pros (メリット)
* **役割の明確化**: `Image` = VRAM（テクスチャメモリ）、`Sprite` = 画面に描画する論理的なビュー、と役割が綺麗に分かれます。
* **メモリ効率が良い**: 大きなアトラス画像から数十個の `Sprite` を生成しても、実体データは1つの `Image` のままであり、メモリを無駄にしません（Cardputerのような限られたRAM環境では極めて重要）。
* **将来の拡張性**: 後から「複数フレームのアニメーション情報」や「デフォルトの当たり判定（コリジョンボックス）」などを `Sprite` 側に持たせる拡張が容易です。

### Cons (デメリット)
* **生成の手間**: 使う側は `img = Image.load(...)` とした後に `spr = Sprite(img, 0, 0, 16, 16)` のように2段階の手順を踏む必要があり、少し冗長になります。
* **Cモジュール化時の参照管理**: Cモジュールで実装する際、`Sprite` 構造体の中に `Image` オブジェクトへのポインタを持たせ、ガベージコレクション（参照カウント）が正しく機能するように気をつける必要があります。

---

## 結論・推奨案

Cardputer（MicroPython環境）において最もネックになるのは **RAM容量** です。
ゲーム開発では「1枚の大きなスプライトシート画像から多数の小さなスプライトを切り出して使う」のが基本パターンになるため、バッファ共有がアーキテクチャレベルで保証されている **Option B（Sprite クラスの新設）を推奨します。**

### 推奨されるAPIデザイン案

APIが冗長になる欠点を補うため、`Image` クラスにヘルパーメソッドを持たせると使いやすくなります。

```python
# エンジン側 (engine/sprite.py)
class Sprite:
    def __init__(self, image, u, v, w, h, colkey=-1, tint=None):
        self.image = image
        self.u = u
        self.v = v
        self.w = w
        self.h = h
        self.colkey = colkey
        self.tint = tint  # 生成時のパレット置換等

# Imageクラスにヘルパーを追加
class Image:
    ...
    def subimage(self, u, v, w, h, colkey=-1):
        return Sprite(self, u, v, w, h, colkey)

# ----- ゲーム側での利用イメージ -----
# 読み込み
atlas = Image.load("images/atlas.uimg")
# スプライトの定義
bike_spr = atlas.subimage(0, 0, 64, 48, colkey=0x0000)
face_spr = atlas.subimage(64, 0, 42, 42, colkey=0x0000)

# 描画 (中心座標ベースで、u,v,w,hは省略できる)
fb.screen.sprite(cx, cy, bike_spr, rotate=0.5, scale=1.0)
```
