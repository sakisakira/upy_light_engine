# プロファイリングから得られたパフォーマンス改善の知見

**調査日時**: 2026年7月4日〜7月5日  
**関連計画**: [plan_30_c_cpp_engine_architecture.md](../plans/plan_30_c_cpp_engine_architecture.md) (Phase 3b〜3c: MicroPython ファームウェア組み込み、ダブルバッファリング及び非同期SPI DMA実装時)

MicroPython搭載の Cardputer (ESP32-S3) 実機において、ゲームエンジン（`upy_light_engine`）を60FPSで動作させるために行った各種プロファイリングと最適化から、今後の開発に役立つ重要な知見が多く得られました。
以下に、システムアーキテクチャやゲームロジック実装におけるベストプラクティスとして記録します。

## 1. ハードウェア・アーキテクチャの知見

### チャンク分割DMA (Chunked DMA) による省メモリ化と並列化
当初、画面全体（240x135 = 約64KB）のダブルバッファリングを試みましたが、ESP32のSRAMの断片化により大きな連続したDMAメモリの確保に失敗（`MemoryError`）しました。
**解決策**: 画面を縦15ラインごとにスライスし、「1つのチャンク（約7.2KB）を描画している間に、もう1つのチャンクをSPI経由でDMA送信する」という**チャンク分割レンダリング**を採用しました。
* **知見**: これにより必要なDMAメモリがわずか **14.4KB** に激減しつつ、メモリ帯域をフル活用する完璧な非同期転送が実現できました。組み込み環境では「巨大なバッファを1つ持つ」よりも「極小のバッファを並行して回す」方が圧倒的に安定し、高速に動作します。

### マルチコア (Core 0 / Core 1) の完全分業
Pythonの実行環境（VM）はESP32の **Core 0** に固定されています。もしSPI転送やピクセル描画（DisplayListの消化）をPythonのスレッドや同期関数で行うと、描画が終わるまでゲームロジックがストップしてしまいます。
**解決策**: C言語拡張モジュール内でFreeRTOSの `xTaskCreatePinnedToCore` を用い、描画専用タスクを **Core 1** に立ち上げました。
* **知見**: プロファイリングの結果、Python側が次フレームの処理を開始する前に呼ぶ待機関数 `sync()` の消費時間がわずか **0.16 ms** になりました。これは、**「Pythonが次の計算をしている裏で、Core 1とハードウェア(DMA)が完全に前のフレームの描画を終えていた」** ことを意味します。この分業により、グラフィックス側のハードウェア制約（16.6ms以内）は完全にクリアされました。

## 2. Python側のボトルネックに関する知見（The True Bottleneck）

Core 1とDMAによる超高速な描画基盤が完成しても、おもちゃのようなサンプル `main.py` のFPSは **45 FPS** 程度にとどまりました。この原因をプロファイラで特定した結果、**「Pythonが最も苦手とする処理」** が浮き彫りになりました。

### 1) Pure Pythonでの細かい反復処理（最大の戦犯）
画面に `FPS: 45` という短いテキストを表示するだけの `draw_text()` 関数に、なんと **4.5 ms** もかかっていました。
* **原因**: C言語側にテキスト一括描画の機能がなかったため、Pythonの `font.py` が「文字列を1文字ずつforループで回し、UV座標を計算し、`push_draw_sprite` を個別に呼ぶ」という処理をしていました。
* **知見**: Pythonインタプリタ上でのループ展開、属性アクセス、そして関数呼び出しのオーバーヘッドは莫大です。このような「1ピクセル単位」や「1文字単位」のミクロな反復処理は絶対にPythonで行わず、C言語の関数（例: `push_draw_text`）に一括で丸投げする（バッチ処理する）必要があります。

### 2) 毎フレームの浮動小数点と三角関数
20個の矩形を波打たせるための `draw_rects()` に **2〜4 ms** かかっていました。
* **原因**: `math.sin` や `math.cos` をループ内で毎フレーム20回呼び出し、float型の演算をしていました。MicroPython環境でのfloat演算は非常に重いです。
* **知見**: まともなゲームを作る際、毎フレーム実行される計算ロジックにfloatの三角関数を使ってはいけません。固定小数点演算（整数計算）や、あらかじめ計算して配列に入れた「サイン波テーブル」を利用することで、計算負荷はゼロに近くなります。

### 3) 魔法の杖: `@micropython.viper`
それでも複雑な計算ループが必要な場合、関数に `@micropython.viper` デコレータをつけることで、型指定付きのネイティブCコードに近い速度で実行させることが可能です。Pythonの遅さはアーキテクチャの工夫や言語機能の活用で十分にカバーできます。

## 3. スレッドとマルチタスクの知見

### GIL (Global Interpreter Lock) の飢餓（Starvation）現象
音楽のMMLシーケンサー（音符を順番に再生する処理）をMicroPythonの `_thread` を用いてバックグラウンドで動かしたところ、**音楽が本来の10倍近く遅く（間延びして）再生される**という致命的な問題が発生しました。
* **原因**: MicroPythonのマルチスレッドは、一つのスレッドしかPythonコードを実行できない「GIL」によって守られています。ゲームのメインループ（Core 0）がCPUを100%近く使い切り（`time.sleep_ms()` での待機時間が0になる状態）、GILを手放さなくなった結果、バックグラウンドの音楽スレッドに実行権が全く回らなくなっていました。
* **知見**: MicroPythonにおいて、計算負荷の高いメインループと並行して `_thread` を走らせることは極めて危険です。解決策として、音楽スレッドを廃止し、メインループの `update()` の中で「経過時間をチェックして次の音符に進める（ポーリング方式）」に変更したところ、遅延は完全に解消しました。状態管理はすべてメインスレッド内で同期的に（ステートマシンとして）処理するのが最も安全で確実です。

---

## 結論

**「ゲーム全体をC言語で書き直す必要は全くない」**

現在の **「描画・音響の重い部分はC言語（Core 1 + DMA）、ゲーム進行やロジックはPython（Core 0）」** というアーキテクチャは、パフォーマンスと開発効率の両立において最適な解です。
今後のゲーム開発やエンジン拡張においては、**「Python側にループ処理やミクロな計算をさせず、Cモジュール側へ大きな塊として仕事を投げる（APIを太くする）」** ことを徹底するだけで、30FPSはおろか60FPSのまともなゲームが十分に動作します。

## 4. メモリアロケーションとGCの隠れた代償（Phase 31 追記）

パフォーマンスチューニング（Plan 31）の過程で、処理速度だけでなく「メモリの動的確保（アロケーション）」が引き起こすガベージコレクション（GC）のスパイクが、FPSを大きく引き下げる要因であることが判明しました。

### 1) ループ内の微小なオブジェクト生成
ゲームループ内で以下のような記述をしていました：
* `text.encode('ascii')` （毎フレーム数回のバイト列オブジェクト生成）
* `bx, by, bw, bh = 50, 40, 40, 40` （タプルのアンパックによるオブジェクト生成）
* `x = sprite.w / 2` （浮動小数点数の割り算によるfloatオブジェクト生成）

* **結果**: これらが塵も積もって「1フレームあたり約280バイト」のメモリリーク（ゴミ）を生み出し、数秒ごとにGCが走って処理をブロックし、結果として全体のFPSが36で頭打ちになっていました。
* **知見**: 毎フレーム呼ばれる `update` や `draw` 内では、極力オブジェクトを生成してはいけません。文字列は最初から `b"TEXT"` として渡し、タプルは `x = 50; y = 40` と分け、割り算は `//`（整数除算）を使うなどの工夫が必要です。

### 2) C言語側のループアンスイッチング（手動展開）
文字描画のC言語側のループにおいて、引数で渡される `tint != -1` の条件分岐をピクセルごとのループ（一番内側）で行っていました。
* **知見**: ESP32のビルドでよく使われる `-Os`（サイズ優先最適化）では、コンパイラがコードサイズ増大を嫌ってループ展開（アンスイッチング）を行わないことがあります。この分岐を「ループの外」に手動で追い出したことと、上記のアロケーション撲滅を組み合わせた結果、**36 FPS から 47 FPS（約28%の速度向上）** という劇的な改善が見られました。塵も積もれば山となります。
* **追記**: その後 `engine_render.c` に対して `-O3` を適用しましたがFPSは47から変化しませんでした。これはC側のピクセル描画ループは手動展開の時点で十分最速に達しており、残りの時間は「SPI転送（約6.5ms）」や「Python側からのAPI呼び出しオーバーヘッド」が占めていることを示しています。
