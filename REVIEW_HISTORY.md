## 2026-06-14: HALクラス構造の分離と `blt` の汎用化

**【指摘内容】**
> 元々はスプライト(ARGB4444)とFramebuffer(RGB565)は別のクラスであると想定してたけど、同じクラスとしてパフォーマンス上問題なければそれでもいいです。その場合はフォーマットを外部から識別できるようプロパティなどを用意して、またbltの引数がどちらでも動くようにした方がいいと思います。microPythonの場合で実装する場合のことも考慮してパフォーマンスについては検討して下さい。
> 
> *(HALを使う側からは「Framebufer」という名前でアクセスできるといいと思います。)*

**【設計判断と対応】**
MicroPython環境におけるパフォーマンスとメモリ効率を考慮し、クラスを完全に分離する設計を採用しました。
- **`Image` クラスの新設**: ARGB4444フォーマットのスプライトデータを保持する専用クラス。MicroPython環境では `framebuf.FrameBuffer` を継承しないただのデータコンテナとすることで、初期化のオーバーヘッドとメモリ消費を最小限に抑止。
- **`Framebuffer` への名称変更**: `PyxelFrameBuffer` から名称を変更。RGB565の画面バッファを管理し、MicroPython環境では標準の `framebuf.FrameBuffer` を継承してネイティブの描画API(`rect`, `fill`等)を利用可能に。
- **`blt` の自動判定化**: 引数に渡されたオブジェクトの `.format` を確認し、`Image` (ARGB4444) であればアルファブレンド合成を、`Framebuffer` (RGB565) であれば `colkey` を使った透過コピーを自動的に行うように修正。

## 2026-06-14: アルファブレンド演算の高速化（ビットシフト化）

**【指摘内容】**
> aは0,15の場合はもう考慮しなくていいから、いっそのこと inv_a = 16 - a として // 16 を >> 4 としたら速くなることってない? もしそうなら検討してくれる? 速くならないのなら現状のままでいいよ。
> 
> *(対象コード: `out_r = (sr * a + dr * inv_a) // 15`)*

**【設計判断と対応】**
非常に有効な最適化テクニックであると判断し、実装に取り入れました。
- 完全に透明(`a=0`)と完全に不透明(`a=15`)のケースは事前に `continue` 等で分岐処理されているため、実際のブレンド処理では `a` は 1〜14 の値をとる。
- したがって、`15` での除算を `16` での除算で近似（`inv_a = 16 - a` とし、`>> 4` のビットシフトを使用）しても、各色成分の最大値スケールは崩れずに正しい結果が得られる。
- **効果**: マイコン環境（MicroPython + Viper）において、比較的重い「除算命令」を排除し、1サイクルの「ビットシフト命令」に置き換えることができたため、描画ループの大幅な高速化が実現した。

## 2026-06-14: Framebufferの `fill`, `rect` 追加と色フォーマットの統一

**【指摘・要望内容】**

**【設計判断と対応】**
ユーザー向けAPIとして色指定を ARGB4444 に統一することで、利用側の負担を減らす設計としました。
- **内部変換メソッドの追加**: `_col4444_to_565(col)` を両HAL実装に追加し、ARGB4444の色情報を即座に内部バッファ用のRGB565に変換する処理を実装。
- **`fill`, `rect` の実装**: ARGB4444 で色を受け取り、内部で変換した上でバッファに書き込む（またはMicroPythonの場合は `super().fill` や `super().rect` に渡す）ように実装。
- **低レイヤーAPIの確保**: 生のRGB565値を直接渡せる `fill_565` と `rect_565` を用意し、パフォーマンスや互換性を重視する用途への逃げ道も確保した。

## 2026-06-14: 一般的なRGB(0-255)色指定ユーティリティの追加

**【指摘・要望内容】**
> ゲーム側に見せる色指定がARGB4444のままだと、利用者が直感的に色を指定しにくい懸念がある。ARGB8888(一般的なRGB)からARGB4444に変換するutility関数を用意してはどうか。

**【設計判断と対応】**
開発者の利便性を向上させるため、直感的な `0-255` ベースの色指定から内部フォーマットの整数を生成するヘルパー関数を実装しました。
- `hal_framebuffer.py` に `color(r, g, b, a=255)` 関数を追加。
- 内部でビットシフト (`>> 4` および `<<`) を行い、瞬時にARGB4444の整数値（例: `0xF00` など）を返す仕様とした。
- これにより、ゲームのロジック側では `hal.color(255, 0, 0)` のように自然に色を指定できるようになりつつ、HAL層の軽快な設計は維持される。
