# プロファイリングと最終限界の証明（Zero-Allocation Viper の検証）

Cardputer Adv実機でのプロファイリングデータをもとに、エンジンのボトルネックを特定し、MicroPython環境下で可能な極限の最適化（Zero-Allocation Viper）を施しました。
その結果から導き出された**「MicroPythonとViperの限界」**と**「次の一手」**についての最終結論をまとめます。

## 1. 初回プロファイリング結果（最適化前）
お客様からご提供いただいたログにより、1フレームあたり **約137ms（約7.3 FPS）** かかっていることが判明しました。

| 処理ブロック | 所要時間 (ms) | 分析 |
| --- | --- | --- |
| `update` | 3.2 ms | 問題なし |
| `draw_rects` | 8.1 ms | 問題なし |
| `draw_sprites` | 18.2 ms | 問題なし（巨大な回転スプライト含む） |
| **`draw_text`** | **54.2 ms** | **最大のボトルネック。1フレームで約120文字を描画。** |
| **`display_show`** | **37.0 ms** | **第2のボトルネック。SPI転送とパレット変換。** |
| `gc` | 6.0 ms | 毎回強制GCしても6ms。 |

**仮説**: `draw_text` が遅いのは、1文字描画する（`blt`を呼ぶ）たびに、11要素の引数タプル `args = (x, y, ...)` を新規生成しており、**1フレームに120回以上のメモリ確保（アロケーション）とGCへの負荷**がオーバーヘッドを生んでいるのではないか？

---

## 2. 究極の最適化（Zero-Allocation Viper）の実装と検証
この仮説を検証するため、C言語モジュール化という「大工事」に逃げる前に、MicroPython環境のままで可能な究極の最適化を実装しました。

### 実装内容
- 初期化時に、再利用可能なC言語レベルの配列 `array.array('i', [0]*16)` を1回だけ確保。
- `blt` や `sprite` の呼び出し時は、タプルを生成せず、この配列の中身を上書きする。
- Viper関数側（`_blt_viper_index8` 等）は、引数をポインタ（`ptr32`）として受け取り、メモリから直接数値を読み出す。

### 第2回 プロファイリング結果（最適化後）
この最適化を適用して再度計測した結果が以下です。

| 処理ブロック | 所要時間 (ms) | 変化 |
| --- | --- | --- |
| `draw_text` | 52.5 ms | ほとんど変わらず！ |
| `display_show` | 37.1 ms | 変わらず |
| **`Free Mem`** | **105056 bytes** | **完全に固定化（GCの負担消滅）！** |

### データが示す「真実」
- **メモリ消費の完全停止**: 第1回のログでは Free Mem が毎フレーム変動していましたが、第2回では `105056` バイトで完全にピタッと止まりました。これは「タプル生成の排除」が大成功し、アロケーション負荷がゼロになったことを意味します。
- **消えない 52.5ms の壁**: GCの負荷が消えたにもかかわらず、`draw_text` の実行時間は縮まりませんでした。これはつまり、**「Pythonのforループで120回配列に数値を詰め込み、PythonからViperの関数を呼び出す」というコンテキストスイッチ（橋渡し）の処理自体**に限界（1文字あたり約0.4ms）がきているということです。
- **SPI転送前の色変換の壁**: `display_show` にかかっている 37ms も、PythonからViperを呼び出して全ピクセルをINDEX8からRGB565へ変換する処理速度の限界です。

---

## 3. 最終結論：Cモジュール化の必要性
「推測するな、計測せよ」の原則に則り、精密なプロファイリングを行った結果、**MicroPython + Viper のアプローチは理論的な限界に到達している**ことが証明されました。

目標である **30FPS**（1フレーム33ms以内）を達成するためには、現在 `draw_text` だけで52ms、`show` だけで37ms かかっている現状を打ち破らなければなりません。

### なぜCモジュールなのか？
1. **関数呼び出しオーバーヘッドの完全排除**
   - テキスト描画の「文字数分のループ」そのものをC言語側に移すことで、120回のPython→Viper呼び出しが「1回のPython→C呼び出し」になり、52msの処理が数msに激減すると推測されます。
2. **メモリの一括変換**
   - 画面バッファ全体の色変換（`show`）をC言語のネイティブポインタ演算で一気に実行することで、37msの処理を10〜15ms程度まで短縮できる見込みです。

今回、推測で闇雲にC言語へ書き直すのではなく、**「データによって限界を証明し、明確な根拠を持ってCモジュール化へ進む」**という非常に価値のある議論と検証を行うことができました。
