# M5Stack Cardputer Adv のメモリ使用状況調査 (2026-07-12)

M5Stack Cardputer Adv (ESP32-S3, PSRAM非搭載) の約328KBの内蔵SRAMに対する、`motorcycle_cardputer` (upy_light_engine) 実行時のメモリ使用状況の調査結果です。

## 全体像 (SRAM 328KB の内訳)

ESP32-S3のSRAMは、大きく分けて「C層（FreeRTOSヒープ / DMAバッファ）」と「MicroPythonヒープ」の2つの領域に分割されて利用されています。

### 1. C層 (FreeRTOSヒープ / DMAバッファ): 約 130KB
MicroPythonからは直接見えないシステムヒープであり、主に `upy_light_engine` のディスプレイ制御等で消費されます。

- **ダブルバッファ (`_c_fbs`)**: **約 126.5KB**
  - 解像度 240x135、色深度 16bit (RGB565 = 2バイト/px)
  - 240 * 135 * 2 = 64,800 バイト/面
  - これをダブルバッファ（2面）確保しているため、`64,800 * 2 = 129,600` バイトを消費。
- **ディスプレイリスト (`_lightengine.DisplayList`)**: **動的**
  - コマンドを積むための配列。C層で動的に `malloc/realloc` されます。
- **SPI DMA オーバーヘッド等**: 数KB

### 2. MicroPython層 (GCヒープ): 約 141KB
MicroPython上でPythonスクリプトが動作するための専用ヒープです。シリアルログの `GC: total: 144832` から、**全体で約 141.4KB** 割り当てられていることが分かります。

**【Titleシーンでの使用内訳 (約 138KB使用 / 3.4KB空き)】**
- **フォントデータ (`font.afnt`)**: **約 13KB**
- **小型フォント (`font_5x7.afnt`)**: **数KB**
- **画像バッファ (`ImageBufferManager`)**: **12KB**
  - 1/4サイズに縮小したことで、35KBから12KBに削減。これにより起動時のクラッシュを回避しています。
- **Pythonオブジェクト群**: **残り全て (約 110KB前後)**
  - モジュール自体のロード (`main.py`, `game.py`, `world.py` 等)
  - クラス・インスタンス (`Title`, スプライト等)
  - MicroPython自体の実行スタック、ローカル変数など

## 空き領域 (Free Memory) の現状と課題

Python側のヒープ状況は非常に逼迫しています。

```text
GC: total: 144832, used: 141504, free: 3328, max new split: 7680
 No. of 1-blocks: 2189, 2-blocks: 174, max blk sz: 1080, max free sz: 72
```

- **実質的な空き容量**: 合計で約11KB（`free: 3328` + `max new split: 7680`）。
- **最大の連続ブロック (`max free sz`)**: 72ブロック × 16バイト = **1,152バイト (1.1KB)**。

### 問題点
空き容量がトータルで11KBあっても、メモリが断片化（フラグメンテーション）しているため、**一度に1.1KBを超えるまとまったメモリ（例えば大きな画像のロードや、巨大なリストの生成）を要求すると即座に `MemoryError` が発生してクラッシュします**。
現状はアセットを1/4サイズに極小化することで「1回あたりのアロケーションサイズ」を強引に1.1KB以下に抑え込んでいますが、非常に薄氷を踏む運用状態と言えます。

## 今後の改善アプローチ
画質改善（元の解像度への復元）と FPS向上（現在10FPS）の両立には、メモリの空き容量を増やす必要があります。

1. **ディスプレイバッファの削減**
   - 126KBものC層SRAMを占有しているダブルバッファを削減（シングルバッファ化や、走査線単位での描画方式への変更）できれば、余ったSRAMをMicroPythonヒープに回すことができます。
2. **PSRAMの活用**
   - Cardputer Adv 以外のデバイスで動作させる場合、PSRAMを有効にしてヒープ全体を外付けメモリに逃がす。
3. **Cモジュール化の促進**
   - `GameGround` など重いクラスや、大量の `Vec2` インスタンスを生成する処理をCモジュールに落とし込むことで、Python側のオブジェクトヘッダ（16バイト/個）の消費を削減します。
