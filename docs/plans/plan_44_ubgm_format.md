# 案C: サウンドアセットのバイナリフォーマット（.ubgm）導入とエンジン改修計画

ESP32等の制約の強いデバイスでの `MemoryError`（巨大なMML文字列のパースによるメモリ枯渇）を根本から解決するため、軽量エンジン `upy_light_engine` に新たなサウンド専用バイナリフォーマット **`.ubgm`** を導入する計画です。

## User Review Required

> [!IMPORTANT]
> この計画では、エンジン側のアーキテクチャ変更（ツール追加とロード関数の追加）を行います。
> ゲーム側（motorcycle_cardputer）の修正は、このエンジン側改修が完了した後に別途行う想定です。
> バイナリフォーマットの仕様やアプローチに問題がないかレビューをお願いします！

## Proposed Changes

### 1. `.ubgm` フォーマットの仕様定義
MMLを事前コンパイルし、メモリに一切の無駄なオブジェクト（タプルやリスト）を生成せずに直接読み込めるバイナリ形式を定義します。

**ヘッダ構造（合計64バイト）**:
- `0x00-0x03`: マジックバイト (`UBGM`)
- `0x04`: バージョン (`0x01`)
- `0x05`: トラック数（最大4）
- `0x10-0x3F`: 各トラックのメタデータ（4トラック分 × 12バイト）
  - `intro_note_count` (2 bytes)
  - `loop_note_count` (2 bytes)
  - `intro_data_offset` (4 bytes)
  - `loop_data_offset` (4 bytes)

**データブロック**:
1音符あたり **6バイト** に圧縮して順次格納します。
- `Frequency (Hz)`: 2 bytes (unsigned short)
- `Duration (ms)`: 2 bytes (unsigned short)
- `Volume`: 1 byte (0-255)
- `Wave Type`: 1 byte (0-255)

※この形式なら、1トラック1000音符でもわずか6KBとなり、ESP32のヒープメモリを全く圧迫しません。

---

### 2. エンジン側ファイルの修正

#### [NEW] `upy_light_engine/tools/mml2ubgm.py`
MML文字列（またはJSON等の元データ）を読み込み、エンジンに内蔵されている `mml_parser.py` を使ってパースしたのち、上記で定義した `.ubgm` バイナリファイルを出力するコンパイラツールを新設します。PC上で開発時に使用します。

#### [MODIFY] `upy_light_engine/engine/sound.py`
新しく以下の関数とクラスを追加します。
- `load_ubgm(filepath)`: `.ubgm` ファイルを `rb` モードで開き、ファイル全体を単一の `bytes`（または `bytearray`）としてメモリに読み込みます。
- `class _UBGMTrack`: リストの代わりに各HALへ渡すためのラッパークラス。`__len__` と `__getitem__` を実装し、インデックスアクセスされた時だけ `struct.unpack_from` でバイナリから6バイトを取り出して `(freq, duration, vol, wave)` のタプルを返す「遅延評価リスト」として振る舞います。
- `play_tracks(intro_tracks, loop_tracks)`: ロード済みのトラック（`_UBGMTrack` のリスト等）をHALに渡して再生を開始するAPIを追加します。

> [!TIP]
> HAL側 (`sound_cpython.py`, `sound_micropython.py`, `sound_wasm.py`) の `play_sequence()` は現在「タプルのリスト」を受け取る前提のコードになっていますが、Pythonのダックタイピングを利用し `_UBGMTrack` クラスがリストと同じように振る舞うため、**HAL側のコードは一切変更せずに済む**という非常にエレガントで安全な設計になっています。

### 3. 新しい `.mml` テキストフォーマットの定義と導入
現在のカンマ区切りのMML表現は可読性が低いため、小節単位（または任意のブロック単位）で複数チャンネルのMMLを記述できる新しいテキストフォーマットを定義し、コンパイラを拡張します。

**フォーマットの仕様**:
- `#` から始まる行はコメントとして無視されます。
- `[` と `]` で囲まれた行（例: `[Measure 1]`）は人間向けの区切り（視認性向上）として無視されます。
- `$` の行は、それ以降が「ループ再生部分」であることを示します。
- `チャンネル番号: MML文字列` の形式で各トラックの音符を記述します。
- 同一チャンネル内で文字列が長くなる場合は、複数行に分割して記述することも可能です（改行は無視され、すべて結合されます）。

**記述例**:
```mml
[Measure 1]
0: T100 O4 V100 C2
1: T100 O4 V100 R2
2: T100 O4 V100 R2

[Measure 2]
0: C4 C4 C4
1: E4 E4 E4
2: R4 G4 G4
```

> [!WARNING] ユーザーレビュー・懸念事項への対応
> **懸念**: 「motorcycle側で使っているBGMが形式上は非常に短かい音符の連続で長い音符を表現してるぽいので、そこでの視認性が本当に確保されるのか」
> **対応**:
> 1. 新しいフォーマットでは**「同一ブロック内での改行」を許可**します。そのため `C8&C8&...` のように長大な文字列になっても適宜改行を入れることで横スクロール地獄を防ぐことができます。
> 2. 根本的な解決として、motorcycle 側の `convert_bgm.py`（JSONからMMLを生成するツール）を改修し、タイ(`&`)の連続を適切な長さの音符（`C2` や `C4.` など）にまとめる最適化処理を追加します。これにより、出力されるMML自体がスッキリと短くなり、高い可読性が確保されます！

### 4. `tools/mml2ubgm.py` の拡張
- コマンドラインから `.mml` テキストファイルを入力として受け取った際、上記の新フォーマットとしてパースする処理を追加します。
- 内部的には、各チャンネルごとに記述されたMML文字列を連結し、最終的に既存の `engine.mml_parser` に流し込んで `.ubgm` を出力する仕組みにします。

## Verification Plan

### 自動・手動テスト
1. **エンジンの修正確認**:
   `upy_light_engine` 側でテスト用スクリプトを作成し、短いMML文字列を `.ubgm` に変換 → `load_ubgm` で読み込み → `play_tracks` でPC上（CPython）で正しく音が鳴るかを確認します。
2. **メモリ使用量の検証**:
   数千ノートの長大なMMLを `.ubgm` に変換し、読み込み時に大量のタプルオブジェクトが生成されず、メモリ使用量が激減していることを確認します。
3. **MMLフォーマットのパース確認**:
   新フォーマットの `.mml` ファイルを作成し、`mml2ubgm.py` が正しく各チャンネルを連結してコンパイルできることを確認します。
