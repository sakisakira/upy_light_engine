# Walkthrough: Plan 31 - Performance Tuning & Memory Leak Fixes

**日付**: 2026年7月5日

このウォークスルーでは、MicroPython環境での `main.py` のパフォーマンスとメモリ効率を改善するために実施した修正内容をまとめます。

## 1. `draw_text` のC言語化（計画の確認）
Plan 31の最重要課題であった「テキスト描画のCモジュール化」ですが、コードを調査した結果、なんと **過去のフェーズ（Phase 3初期）ですでに `dl_push_draw_text` としてCエンジン側に実装・結合済み** であることが判明しました。
プロファイリング時に `draw_text` で4.5msかかっていた本当の理由は、Python側のループではなく、描画のたびに `text.encode('ascii')` を呼び出して巨大な動的アロケーション（後述のメモリリーク）を引き起こしていたことでした。今回はC言語側の修正は行わず、Python側からの呼び出しの最適化のみで解決しました。

## 2. フォント読み込みエラーの解消（メモリ断片化の回避）
MicroPythonのヒープメモリにおいて、大きなリスト（LUT等）を先に生成するとメモリが「断片化」し、その後で25KBなどの連続した領域（フォントデータ用）を確保しようとした際に `MemoryError` が発生する問題がありました。
**対応:** 
- `main.py` の冒頭（LUTを生成する前）で `font_lib.Font` を呼び出し、綺麗なメモリ状態のうちにフォントのバイト配列を確保するように順序を変更しました。

## 3. 浮動小数点計算の削減（LUT化）
ゲームループ内で `math.sin` を呼び出すと計算が非常に遅いため、事前に計算済みの値を配列に保存して参照する方式（ルックアップテーブル / LUT）に変更しました。
**対応:**
- 当初は `array.array('f')` や `list` を試みましたが、要素アクセス時に動的にfloatオブジェクトが生成（アロケーション）されてしまう問題がありました。
- 最終的に `tuple` を使用してテーブルを構築することで、初期化時にfloatオブジェクトを確保し、ループ内ではその参照を返すだけ（アロケーションゼロ）の構成に落ち着きました。

## 4. ゲームループ内のメモリリーク撲滅
ループ中にオブジェクトが動的生成されると、ガベージコレクション(GC)が頻繁に発生しFPSの低下を招きます。今回、1フレームあたり数百バイト漏れていた原因を特定し、排除しました。
**対応箇所:**
- **タプルのアンパック:** `bx, by, bw, bh = 50 + i*30, ...` はタプルオブジェクトを生成するため、`bx = ...`, `by = ...` と個別に代入するように変更。
- **浮動小数点数の割り算:** `sprite.w / 2` は float を生成するため、`sprite.w // 2`（整数除算）に変更。
- **文字列のエンコード:** `text.encode('ascii')` は毎フレーム `bytes` オブジェクトを生成するため、`font_lib.text()` 等に最初から `b"SCORE 1234"` といったバイト列を渡せるようにAPIを修正し、`main.py` 内の固定文字列を `b"..."` で定義しました。

## 5. パーティクルシステム（Viperデモ）の削除
`@micropython.viper` を用いたパーティクル計算処理を追加しましたが、計算自体は高速でも、結果を描画するためにPython側から `fb.screen.pset` を100回連続で呼び出す（C言語とのバウンダリを超える）コストが非常に高く、かえってFPSを36まで低下させる原因となっていました。
**結論:**
Pythonで細かい描画コマンドを大量に発行するのは避け、ロジックのみを扱い、描画は大きな塊でCモジュールに委譲するという設計思想に従い、このデモコードは削除しました。
