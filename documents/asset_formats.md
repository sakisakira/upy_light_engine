# UPY Light Game Engine Asset Formats

このドキュメントでは、本ゲームエンジン（upy_light_engine）で使用される軽量な独自アセットフォーマット（`.uimg`, `.afnt`）のバイナリ構造について説明します。

MicroPythonの限られたメモリとCPUリソースで高速に画像合成（アルファブレンド等）を行うため、ファイルフォーマットはあえてパースしやすい単純なバイナリ構造と「ARGB4444 (16-bit)」ピクセルフォーマットを採用しています。

---

## ピクセルフォーマット: ARGB4444

両フォーマット共通で、画像データは **ARGB4444（1ピクセルあたり16ビット/2バイト）** のリトルエンディアン形式で格納されます。
1バイト（8ビット）ずつではなく、16ビットを以下の構成で分割しています。

| Bits 15-12 | Bits 11-8 | Bits 7-4 | Bits 3-0 |
| :---: | :---: | :---: | :---: |
| Alpha (A) | Red (R) | Green (G) | Blue (B) |

> **Endianness（エンディアン）の注意点**:  
> ファイルにはリトルエンディアンで保存されます（例: `0x1234` なら `0x34` `0x12` の順）。  
> `hal/framebuffer_micropython.py` や `hal/font.py` でファイルから `f.read()` で読み込んだ `bytearray` は、Viper側で `ptr16` としてキャストされることで、プログラム上では16ビット整数としてそのまま直接扱われます。

---

## 1. UIMG 形式 (`.uimg`)

単一のスプライトや背景画像を格納するためのフォーマットです。
`tools/png2uimg.py` によってPNGから変換されます。

### ヘッダ構造 (計10バイト)

| Offset | サイズ | 型 | 説明 |
| :--- | :--- | :--- | :--- |
| `0x00` | 4 bytes | ASCII | マジックナンバー: `"UIMG"` |
| `0x04` | 1 byte | uint8 | Version: 現在は `1` |
| `0x05` | 1 byte | uint8 | Format: `1` (1=ARGB4444) |
| `0x06` | 2 bytes | uint16 (LE) | Width: 画像の幅 (ピクセル数) |
| `0x08` | 2 bytes | uint16 (LE) | Height: 画像の高さ (ピクセル数) |

### データ構造

ヘッダに続いて、画像の左上から右下に向かってピクセルデータ（1ピクセルあたり2バイトのARGB4444値）が `Width * Height` 個格納されます。

---

## 2. AFNT 形式 (`.afnt`)

アトラス型のビットマップフォントを格納するためのフォーマットです。
`tools/font_converter.py` によってTTF/BDF/PNGから変換されます。
現在は「AFNT (v1)」と「AFN2 (v2)」の2つのバージョンが共存しています。

### AFNT (v1) - 標準ASCIIフォーマット
ASCIIコード（0x20〜0x7E）の連続した95文字を固定で保持する形式です。

#### ヘッダ構造 (計8バイト)

| Offset | サイズ | 型 | 説明 |
| :--- | :--- | :--- | :--- |
| `0x00` | 4 bytes | ASCII | マジックナンバー: `"AFNT"` |
| `0x04` | 1 byte | uint8 | Char Width: 1文字あたりの幅 (px) |
| `0x05` | 1 byte | uint8 | Char Height: 1文字あたりの高さ (px) |
| `0x06` | 1 byte | uint8 | Columns: アトラス画像の横の文字数 |
| `0x07` | 1 byte | uint8 | Rows: アトラス画像の縦の文字数 |

※ ピクセルデータの合計サイズは `(Char Width * Columns) * (Char Height * Rows) * 2` バイトになります。

---

### AFN2 (v2) - カスタムキャラクターマップ形式
「ゲームに登場する特定の文字だけ」を抽出し、メモリ消費を大幅に削減するために導入された新しいフォーマットです。
ファイル内に「どのグリフ（画像）が、どの文字コードに該当するか」のメタデータ（キャラクターマップ）を保持します。

#### ヘッダ構造 (可変長)

| Offset | サイズ | 型 | 説明 |
| :--- | :--- | :--- | :--- |
| `0x00` | 4 bytes | ASCII | マジックナンバー: `"AFN2"` |
| `0x04` | 1 byte | uint8 | Char Width: 1文字あたりの幅 (px) |
| `0x05` | 1 byte | uint8 | Char Height: 1文字あたりの高さ (px) |
| `0x06` | 1 byte | uint8 | Columns: アトラス画像の横の文字数 |
| `0x07` | 1 byte | uint8 | Rows: アトラス画像の縦の文字数 |
| `0x08` | 2 bytes | uint16 (LE) | Num Chars: 収録されている文字の総数 (`N`) |

#### キャラクターマップ構造 (計 `N * 2` バイト)

ヘッダの直後に、収録されている文字の Unicode コードポイント（UTF-16）が `Num Chars` 個だけ連続して並びます。

| Offset | サイズ | 型 | 説明 |
| :--- | :--- | :--- | :--- |
| `0x0A` | 2 bytes | uint16 (LE) | 1番目の文字のコードポイント（例: `0x0041` = 'A'） |
| `0x0C` | 2 bytes | uint16 (LE) | 2番目の文字のコードポイント |
| ... | ... | ... | ... |

*MicroPython側のロード処理 (`hal/font.py`) では、この配列を読み取って `コードポイント -> アトラス上のインデックス(0〜N-1)` に変換する辞書（Dictionary）をメモリ上に構築し、描画時の高速な位置計算に利用します。*

### データ構造 (v1, v2共通)
ヘッダおよび（AFN2の場合は）キャラクターマップに続いて、アトラス全体のピクセルデータ（ARGB4444）が格納されます。
アトラス画像は、左上から右下に向かって左詰め（横 `Columns` 文字、縦 `Rows` 文字）でグリフが配置された1枚の巨大な画像として記録されています。
