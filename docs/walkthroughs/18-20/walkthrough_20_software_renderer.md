# ソフトウェアレンダラーの共通化リファクタリング

## 概要
CPython, WASM, MicroPythonの3つの `Framebuffer` HALクラスにコピー＆ペーストで実装されていた `blt` と `sprite` の描画ロジックを抽出し、共通の描画関数として統一しました。
このリファクタリングにより、描画処理の保守性が向上し、将来的な最適化を1箇所で行えるようになりました。

## 変更内容
1. **`engine/hal/software_renderer.py` の新設**
   * `draw_blt` および `draw_sprite` メソッドを実装しました。
   * MicroPython特有の「ビッグエンディアンでのバッファ直接書き込み」に対応するため、`byte_swap` という引数を導入しました。
   * ループの最内側（inner loop）での分岐を避け、ループの外側で `byte_swap` の真偽による分岐を行うことで、Python特有の関数呼び出しオーバーヘッドや分岐ペナルティを最小限に抑えています。

2. **`sprite()` メソッドの最適化**
   * `sprite()` に渡されるソース画像は `Sprite` オブジェクトを経由しており、必ず `ARGB4444` フォーマットであることが保証されているため、ソースがRGB565であるケースを想定した不要な分岐処理を丸ごと削除しました。これによりコードがシンプル化し、処理効率もわずかに向上しています。

3. **各HALからの委譲**
   * `framebuffer_cpython.py` と `framebuffer_wasm.py` では、`blt` と `sprite` を `software_renderer` 呼び出しに置き換えました（`byte_swap=False`）。
   * `framebuffer_micropython.py` では、`sprite` を `software_renderer` 呼び出しに置き換えました（`byte_swap=True`）。`blt` については、既存の `@micropython.viper` による高速化の恩恵を残すため、今回は変更していません。

## テスト結果
* エンジンのテストベッドである `main.py` を実行し、リファクタリング前と全く同じようにブレンド描画やスプライトの縮小・拡大（端の欠けなく）が正常に機能していることを確認しました。
* パフォーマンスの低下も生じておらず、スムーズに動作しています。
