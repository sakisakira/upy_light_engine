# パフォーマンス・プロファイリングと最適化の洞察 (60 FPS達成の記録)

このドキュメントは、Cardputerの `upy_light_engine` を最適化し、安定した 60 FPS を達成するために行った手順をまとめたものです。当初、C言語コアの描画は高速であったにもかかわらず、エンジンは 42〜45 FPS 程度で頭打ちになっていました。そこで、体系的にプロファイリングを行い、以下のボトルネックを特定して解決しました。

## 1. フレームタイミングとOSティックの丸め誤差
**当初の問題:** `engine/hal/framebuffer_micropython.py` のフレームレート制限ループでは、待機に `time.sleep_ms(sleep_ms)` を使用していました。
**洞察:** FreeRTOS上のMicroPythonにおける `time.sleep_ms` は、CPUの実行権をOSに返還しますが、OSスケジューラのティック精度（ESP32では通常 10ms）に依存するため、指定したスリープ時間がランダムに最大 10ms も超過してしまう問題がありました。
**解決策:** `time.sleep_ms` を高精度なビジーウェイトのループ (`while time.ticks_diff(time.ticks_ms(), t0) < target_ms: pass`) に置き換えました。

## 2. SPI DMA における Python と C の境界オーバーヘッド (と思いきやの偶然の発見)
**当初の問題:** ST7789 ディスプレイの描画ウィンドウを設定する `set_window` 関数が、Python 側の毎フレームの描画ループ内で呼ばれていました。
**洞察:** プロファイリングの結果、この `set_window` が毎フレーム **約 2.0 ms** を消費していました。これをC言語側にハードコードで移植しようと考え、Pythonのループから `set_window` を削除しました。
**奇跡の発見:** 実は**ファームウェアを再ビルド・書き込みしていなかった**ため、C言語側のパッチは一切動いていませんでした！しかし画面は正常に描画され、遅延だけが 0 ms に消滅しました。理由は「**ST7789チップは一度設定されたウィンドウ座標を記憶するため、毎フレーム再設定する必要は全くなかった**」からです。初期化時に1回だけ `set_window` を呼べば十分であり、毎フレームの無駄なSPI転送（2.0 ms）を丸ごと削り落とすことに成功しました。

## 3. 非同期 Core 1 稼働による SRAM バスのストール
**当初の問題:** `_lightengine.submit_and_send()` の呼び出しになぜか **約 5.1 ms** もかかっていました。
**洞察:** キューにプッシュした瞬間、Core 1 の `render_task` が目を覚まし、最適化されたC言語ループを使って SRAM 上のフレームバッファへ猛烈な勢いで描画データを書き込み始めます。ESP32の内部SRAMのメモリバスは両コアで共有されているため、Core 1 のアグレッシブなメモリアクセスによりメモリコントローラが飽和し、Core 0（MicroPythonを実行しているコア）の命令フェッチが待たされる（ストールする）という現象が起きていました。
**解決策:** この物理的なハードウェアストールそのものを回避することは困難ですが、このストールを許容した上でも全体の処理時間が 16.6 ms の予算に収まるように、純粋なPython処理（Core 0）を極限まで高速化することにしました。

## 4. MicroPython インタプリタの実行速度の限界
**当初の問題:** `update()` と `draw_all()` という純粋なPythonロジックの実行に **約 14.5 ms** かかっていました。
**洞察:** 160MHz（MicroPythonのデフォルトの起動周波数としてよくある値）では、Pythonのバイトコード実行や属性へのアクセスなどがどうしても遅くなります。
**解決策:** `main.py` の先頭に `machine.freq(240000000)` を追加し、ESP32のCPUを強制的に最大周波数の 240MHz で動作させるようにしました。これによりPythonの実行速度が即座に 1.5倍 に跳ね上がり、`update` と `draw` の合計時間は 約 9 ms にまで短縮されました。

## 最終的な成果
毎フレームの不要な `set_window` の削除、高精度なビジーウェイトによるフレーム制御、そして 240MHz のフルパワー駆動を組み合わせたことで、（Core 1 によるSRAMストールを含めても）トータルのフレーム処理時間は 16.6 ms を大きく下回るようになり、ディスプレイ上で安定して **60-63 FPS** を叩き出すことに成功しました。
