# Cardputer メモリ限界調査と 8-bit アーキテクチャ移行の必要性

## 1. 背景
Cardputer Adv (M5StampS3) 実機において、ゲームエンジン起動時に `MemoryError: memory allocation failed, allocating 64800 bytes` が発生する問題の根本原因を調査した。

## 2. デバイスのハードウェア制約 (PSRAM非搭載)
Cardputer に搭載されている ESP32-S3 (M5StampS3) は **PSRAM (拡張RAM) を搭載していない**。
そのため、MicroPython が利用できるヒープ領域は、ESP32-S3 の内部 SRAM に限定される。

現在のファームウェアビルド設定（`BOARD_VARIANT=SPIRAM_OCT`）では、存在しない PSRAM の管理領域として内部 SRAM が予約されてしまうため、MicroPython のヒープが **約64KB** という極小サイズに制限されていた。
これを修正（`SPIRAM_OCT` を削除）することで、ヒープ領域は **約130KB** に回復する。

## 3. 16-bit (ARGB4444) 継続時のメモリ枯渇の証明
ヒープを約130KBに回復させた上で、現状の 16-bit カラーアーキテクチャ (ARGB4444) を維持できるかを計算した。
メモリ節約のため「シーンごとに画像をロード・破棄する」という理想的なメモリ管理を実装したと仮定し、最も重い「ゲームプレイ画面」の消費RAMを算出する。

### ゲームプレイ画面での要求RAM
1. **フレームバッファ (16-bit)**
   - `240 x 135 ピクセル x 2 バイト = 64.8 KB`
2. **ゲーム中の画像群 (16-bit)**
   - `chara_body.uimg`: 24.5 KB
   - `chara_symbol.uimg`: 24.5 KB
   - `faces.uimg`: 21.6 KB
   - `bike_body.uimg`: 6.1 KB
   - `tire.uimg`: 6.1 KB
   - 合計: **82.8 KB**

**合計要求RAM: 64.8 KB + 82.8 KB = 147.6 KB**

### 結論
物理的な最大ヒープである **130 KB** に対し、**147.6 KB** を要求するため、17.6KBのオーバーフローが確定する。これにPythonのシステムメモリや変数の消費が加わるため、**ARGB4444のままでは物理的に実行不可能 (100% MemoryError が発生する)** ことが証明された。

これを回避する唯一の手段は「画像をファームウェアに Frozen Module として焼き込み、RAM消費を 0 にする」ことだが、これは「ゲームごとにファームウェアをビルドする必要がある」という点で、汎用ゲームエンジンとしての要件を満たさない。

## 4. 8-bit パレットアーキテクチャによる解決
画像とフレームバッファを共に 1ピクセル＝1バイトの **8-bit パレットインデックス方式 (最大256色)** に移行した場合の計算を行う。

### ゲームプレイ画面での要求RAM (8-bit)
1. **フレームバッファ (8-bit)**
   - `240 x 135 ピクセル x 1 バイト = 32.4 KB`
2. **ゲーム中の画像群 (8-bit)**
   - 上記画像サイズの厳密に半分となるため、合計: **41.4 KB**

**合計要求RAM: 32.4 KB + 41.4 KB = 73.8 KB**

### 結論
130 KB のヒープ領域に対し 73.8 KB となり、約 56 KB の余裕が生まれる。
Pythonのコード実行や動的な変数確保に十分な空き容量となる。

また、元となった Pyxel アセットが使用している色数は最大でも200色程度であるため、256色のパレットに収めることで画質の劣化（色落ち）も発生しない。

したがって、汎用エンジンの設計（ゲームスクリプトとアセットを実機にコピーするだけで動く）を維持しつつ Cardputer で動作させるための **唯一の最適解は 8-bit パレットアーキテクチャへの移行である** と結論付けた。
