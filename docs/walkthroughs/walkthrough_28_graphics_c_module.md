# Walkthrough 28: C Module Graphics Optimization Journey

## Background
The Python-only `framebuffer` implementation maxed out at roughly 6-8 FPS (約125ms〜166ms/フレーム). 
プロファイリングの結果、テキスト描画やスプライト描画、ディスプレイ転送に多大な時間がかかっていることが判明しました。
最終的に **36 FPS（約27.7ms/フレーム）** まで最適化することに成功しましたが、その過程で削減した **「約108ms/フレーム」** の内訳と寄与度（ボトルネックの正体）を、ログの推移に基づいて分析します。

---

## 削減時間（約108ms）の内訳と寄与率の分析

最適化前の平均フレーム処理時間を約135msとし、各施策によって何ms削減できたか（＝合計108msに対する寄与率）を以下にまとめます。

### 1位: `fb.color()` のキャッシュ化（削減: 50.1ms / 寄与率: 46.4%）
ユーザーの推測通り、**これが最大のボトルネックでした。**
RGBからパレットインデックスを検索する `fb.color()` は、内部で純粋なPythonループを最大255回（距離計算）回していました。これが1フレームに複数回呼ばれることで、甚大な遅延を引き起こしていました。
キャッシュ辞書 `_color_cache` を導入した直後のログ比較により、以下の削減が明確に証明されています：
- `draw_rects`: 8.35ms → 3.03ms (**-5.32ms**)
- `draw_sprites`: 14.93ms → 4.71ms (**-10.22ms**)
- `draw_text`: 40.50ms → 5.92ms (**-34.58ms**)
**【ログによる証明】**: キャッシュ導入前後のプロファイルログで、上記3つのセクションの合計が **50.1ms** 削減されたことが確認できます。

### 2位: USBシリアル通信（`print`）のオーバーヘッド削減（削減: 17.0ms / 寄与率: 15.7%）
プロファイラが毎フレーム `[PROFILE] ...` を6行プリントしていましたが、USB CDC UARTのバッファ待ちにより、処理がブロックされていました。
**【ログによる証明】**: FPSが14（実測フレームタイム約71ms）と表示されていた時点で、プロファイラの各項目の合計時間は54.1msでした。この差分 **17ms** が、プリント出力による「見えない遅延」であることが判明しました。

### 3位: `convert_palette_chunk` のCモジュール化（削減: 16.3ms / 寄与率: 15.1%）
ST7789への送信前に、INDEX8のピクセル配列をRGB565（16bit）に変換する処理です。
**【ログによる証明】**: 初期Python実装では `display_show: 37.12ms` かかっていましたが、C関数に置き換えた直後のログで `20.69ms` まで低下しました（**-16.43ms**）。

### 4位: SPI転送の80MHzオーバークロック（削減: 約6.5ms / 寄与率: 6.0%）※ここは推察
`display_show`（20.69ms）のうち、C関数での変換は1ms未満であり、残り約20msはSPI転送自体（40MHz）が占めていました。
**【ここは推察】**: 80MHzに倍速化したことで、理論上SPI転送時間は半減（約13ms→6.5ms）したと推察されます。プロファイラをオフにした後に36FPS（全体で約27ms）に到達したことから、この削減が大きく寄与していると考えられます。

### 5位: `gc.collect()` のループ内強制実行の削除（削減: 6.0ms / 寄与率: 5.6%）
毎フレーム手動でガベージコレクションを走らせていました。
**【ログによる証明】**: 全てのプロファイルログにおいて `gc: 5.95ms 〜 6.13ms` と記録されており、これを削除したことで確実かつ安定して **6.0ms** の削減となりました。

### 6位: `draw_text` のC化 ＆ `measure_text` の最適化（削減: 6.0ms / 寄与率: 5.6%）
Pythonでの1文字ずつのループ処理や、辞書引きをCに移行しました。
**【ログによる証明】**: 最初のC化によって `draw_text: 52.41ms` が `46.48ms` に下がり（**-5.93ms**）、その後の最適化で微細な削減が積み重なりました。残る約40msは1位の `fb.color()` だったため、純粋なテキスト配置自体のオーバーヘッド削減幅はこの程度でした。

### 7位: `draw_sprite` のCモジュール化（削減: 4.4ms / 寄与率: 4.1%）
スプライトのアルファブレンドや透過色（colorkey）判定をCに移行しました。
**【ログによる証明】**: 初期Python実装 `19.39ms` から、C関数置き換え直後のログで `15.06ms` へ低下しました（**-4.33ms**）。

### 8位: Cコンパイラの `-O3` 最適化（削減: 約1.5ms / 寄与率: 1.4%）※ここは推察
**【ここは推察】**: デフォルトの `-Os`（サイズ最適化）から `-O3`（速度優先）に変更しました。C側のループ（スプライト描画やパレット変換）がメモリレイテンシをより効率的に隠蔽するようになり、全体で1〜2ms程度底上げされたと推察されます。

---

## 結論
全ての最適化は無駄ではありませんでしたが、ご指摘の通り **`fb.color()` だけで最適化全体の約半分の効果（46%）** を占めていました。
「Cモジュール化」という重いアプローチは確かに必須（15%＋4%＋5%）でしたが、**「隠れたPythonの高負荷ループ（`fb.color`、`print`、`gc`）」** がそれ以上に甚大なボトルネックになっていたことが、今回のプロファイリングログから見事に証明されました。

MicroPythonにおけるゲーム開発では、Cモジュールという「武器」だけでなく、Pythonレイヤーの「無駄な繰り返し計算のキャッシュ化（Memoization）」が何よりも重要であるという強力な教訓が得られました。
