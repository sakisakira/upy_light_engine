# 実装計画

今すぐに実装して欲しい訳ではないけど、将来的にこれらを見据えて作業をして欲しいことを列挙します。
「こうした方がいいのではないか」という意見があれば言って下さい。

* バグ疑い
  * psetでのクラッシュの可能性を調査し、必要であれば修正する。
  * **リソース解放時の Use-After-Free 対策（抽象化の漏れの解消）**:
    * 現在はPythonレイヤーで `image.unload()` を呼ぶ前に手動で `_lightengine.sync()` を呼び、バックグラウンド描画中のリソース解放（パニック）を防いでいる。これをエンジン側で隠蔽し、Python側に低レイヤーを意識させない設計にする。
    * 解決案1: `unload()` 呼び出し時に、Cモジュール内で描画完了まで自動でブロック（同期）してから `free` する。
    * 解決案2: 「遅延解放（Deferred Free）」。`unload()` 時には解放予約フラグのみ立ててPythonをブロックせず、描画完了後にエンジンが安全に `free` する。ただし、ESP32のようなメモリに余裕がない環境では、解放が遅れることで「次のシーンの画像ロード」が一瞬メモリ不足に陥る危険性があるため注意が必要。
  * **シーン遷移時のメモリ管理補助メカニズム（Scene Manager 等）**:
    * 現在、ゲーム側でシーン遷移する際に、`sync()`, `unload()`, `gc.collect()`, `buffer_manager.reset()` などの泥臭い処理をゲームロジックの中に直接書かざるを得なくなっている。
    * 全てをエンジンで自動管理するのは難しくとも、`onStageChanged`（仮称）のようなコールバックやフック、または一連のクリーンアップ処理をラップする補助APIを用意し、ゲーム側のコードを直感的にする。
  * **Python側から利用できる安全なアサーション（クラッシュ）機能の提供**:
    * 現在 MicroPython 環境（Cardputer等）では、想定外のステートで `raise` したり未処理の例外が発生したりすると、画面がフリーズしてリセットボタンを押すしかなくなるケースがある。そのため、ゲーム側でわざと例外を握り潰すフェイルセーフが横行している。
    * これを防ぐため、エンジン側から `engine.assert_(condition, message)` のような機能を提供し、アサーション失敗時にはゲームループを安全に停止した上で、画面（またはコンソール）にエラー情報を表示する仕組みが必要。
  
* HAL
  * ターゲットはCardputer Adv, PC(Windows, macOS), Web (WASM)とします。
* 文字描画
  * 描画色を引数から指定できるようにする。
  * 影付き文字などの重ね描きパターンはあらかじめアトラス画像側に合成・収録しておき、1回の `blt` で高速に描画できるようにする。
    * （フォント変換ツール側の拡張と、エンジン側でのスタイル/オフセット指定の仕組みを追加する）
* フォント
  * 用途に応じたフォントフォーマットの明示的な分離（AFN1とAFN8）を実装する。
    * AFN1: 影なし単色用（1ピクセル=1ビット）。高圧縮でtint機能と相性が良い。動的な影付け（text_shadowed）と併用。
    * AFN8: 影や縁取りなどのスタイル焼き込み済み用（1ピクセル=1バイト）。複数色を保持し、tintを使わず1回のbltで最速描画する用途。
* サウンド
  * 将来的に波形エディタなどを実装する際でも、全プラットフォームでバイトレベルで完全に同一の音を鳴らせるようにする。
* 入力
  * Cardputer Adv上の好きなキーをボタンに割り当てられる機能が欲しい。
  * PC側はキーとゲームコントローラに対応できると良い。(オプション)
  * **マウスおよびタッチパネル入力のサポート**:
    * 画面上の仮想ボタン（UI）をクリック・タップで操作できるようにするため、`mouse_x`, `mouse_y`, `mouse_button` などのポインティングデバイスの座標と状態を取得できるAPIをエンジンに追加する。
      * Cardputer Advではマウス・タッチパネル入力を無視する。

* テストの一時的なファイルは 「tests/tmp」 ディレクトリに置くようにして下さい。
* アサーション。リリース時に実行速度に悪影響を与えないようにしつつ、開発時には問題発覚の機会を増やしたい。
* パフォーマンス
  * ボトルネック箇所があればそれを特定する. プラットフォームごとに異なる可能性もある.
  * パフォーマンスが期待通りでない場合にはプロファイリングを実施する。 
  * **オフスクリーン・レンダリング（Render-to-Texture）APIの提供** (optional)
    * 背景や大量の静的オブジェクトを一度別の `Framebuffer` に事前描画（`submit`時パレットなし）しておき、1枚の巨大な `Image`（スプライト）としてキャッシュ・再利用できる仕組みをPythonゲームレイヤーに公開する。これにより `motorcycle` のような背景負荷の高いゲームのFPSを劇的に向上できる。
    * ※ただし、フルスクリーン(240x135)のバッファを余分に確保すると約32KBのメモリを消費するため、メモリ枯渇への配慮が必要。全画面ではなく「更新頻度の低い空の部分だけ（240x60等）」など、部分的なオフスクリーンバッファを作れる柔軟なAPI設計を検討する。

# 内部設計

* 管理を意識した設計と実装
  * プラットフォーム間をまたいだコード
     * 可能であればプラットフォーム間でコード(Python, C)を共有する。
* パフォーマンスと可用性の両立
  * エンジン機能はCore 1で、ゲームはCore 0で動かせるようにする。
  * ゲーム側からそれを意識させないようなAPIであると嬉しいが、パフォーマンス上無理であればAPI変更もやむなしとする。
  * **[Optional] Native C Module (.mpy) アーキテクチャの導入検討**
    * ファームウェアのビルドを回避しつつ高速なイテレーションを維持するため、ゲームロジック部分をC言語で記述し `mpy-cross / dynruntime` で `.mpy` バイナリとしてコンパイルして動的ロードする構成をオプションとして検討する（Pythonは薄いラッパーとして機能させる）。
