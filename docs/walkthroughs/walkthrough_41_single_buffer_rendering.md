# Walkthrough 41: シングルバッファ化とFreeRTOSヒープの活用

## 概要
Cardputer（ESP32-S3）の限られたSRAM環境において、フレームバッファをダブルバッファからシングルバッファへと削減し、解放されたFreeRTOSヒープ領域をPythonから直接利用できるようにするための修正を行いました。

## 実装内容

### 1. フレームバッファのシングルバッファ化
`engine/hal/framebuffer_micropython.py` におけるCモジュール（`_lightengine`）とのインターフェースを修正し、フレームバッファの確保を2面（ダブルバッファ）から1面（シングルバッファ）へ削減しました。
- 従来は `self._c_fbs` として配列で2つの `_lightengine.Framebuffer` を保持・スワップしていましたが、`self._c_fb` 1つのバッファに固定し、常にこのバッファへ描画・転送を行うように変更しました。
- この変更により、ディスプレイリストの構築先は依然としてダブルバッファリング（`dl_idx` のスワップ）で並列化を維持しつつも、物理的なピクセルバッファは1面のみとなり、理論値通り **約 32.4 KB の FreeRTOSヒープ（SRAM）を解放** することに成功しました。

### 2. Cモジュールへの malloc/free API 追加
解放されたFreeRTOSヒープをPython（ゲームスクリプト）側から有効活用するため、Cエンジンモジュールに以下のAPIを追加しました。
- **`_lightengine.malloc(size)`**: FreeRTOSのヒープから指定バイト数のメモリを確保し、Pythonの `bytearray` として参照渡しで返却する機能。
- **`_lightengine.free(buffer)`**: 上記で確保したメモリを安全に解放する機能。
- **`_lightengine.get_free_heap()`**: 現在のFreeRTOSヒープの空き容量を取得し、デバッグやメモリ監視に役立てる機能。

## 成果と効果
- **メモリ解放の成功**: ダブルバッファ稼働時のFreeRTOSヒープの空きが約 24.8 KB であったのに対し、シングルバッファ化後は **約 57.2 KB** へと増加しました。
- **パフォーマンスの維持**: ティアリング等の描画の破綻や大幅な遅延は確認されておらず、**63 FPS** 程度を維持してゲームが稼働することが実証されました。
- これにより、MicroPythonのガベージコレクション領域を逼迫させることなく、巨大な画像バッファ（ImageBufferManager）をC層から直接確保する基盤が完成しました。
