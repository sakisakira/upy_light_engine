# Walkthrough: エンジンの安全性向上とリファクタリング (Plan 40)
**Date**: 2026-07-12 12:40:21

本ウォークスルーでは、巨大な座標が渡された際に発生していたメモリ破壊（クラッシュ）バグの根本原因の解決と、それに伴うエンジンのアーキテクチャ改善について記録します。

## 根本原因: 整数型のラップアラウンド

`fb.screen.pset()` などに対して画面外の極端に大きな座標（例: `70000`）を渡した際に発生するランダムなクラッシュや描画の乱れは、整数型の仕様によるものでした。

1. Pythonの `int` 型は多倍長整数であり、どんなに巨大な値でも保持できます。
2. 一方、CエンジンのコアAPI（`dl_push_*`）は、メモリ節約のため引数として `int16_t` (16ビット符号付き整数) を受け取っていました。
3. Pythonから巨大な座標が渡されると、MicroPython側で強制的に `int16_t` にキャストされ、**ラップアラウンド（桁あふれによる負数等への変化）** が発生します。その結果、「本来は画面の遥か外側にあるはずの座標」が「画面内の有効な座標」に化けてしまい、クリッピング判定をすり抜けて不正なメモリアクセスを引き起こしていました。

## アーキテクチャの改善: 責任の分離 (Separation of Concerns)

当初はC言語のコアAPI (`dl_push_*`) の引数を `int32_t` に変更し、その中で画面外判定（カリング）を行うアプローチを試みました。しかし、これは「純粋に描画コマンドをバッファに積む」というコアAPIの責務を超えた、場当たり的な設計でした。

最終的に、以下のように責務を完全に分離する美しいアーキテクチャへと改修しました。

### 1. コアAPI (`engine_types.c`) の純化
- `dl_push_*` 関数の引数をすべて本来の `int16_t` に戻しました。
- 内部で行っていた画面外判定やクリッピングのロジックをすべて削除し、「与えられたコマンドを純粋に DisplayList に登録するだけ」の機能に戻しました。

### 2. Pythonバインディング層 (`modlightengine.c`) での保護
- 画面外判定（`is_visible`）と座標の安全化（`sanitize_x`, `sanitize_y`）のロジックを、Pythonとの境界であるバインディング層に移動させました。
- **処理フロー**:
  1. Pythonから渡された引数を、まずラップアラウンドしないよう十分な大きさの `int32_t` で受け取ります。
  2. 完全に画面外であれば、この時点で直ちに `return mp_const_none;` とし、コマンドリストの枠を無駄に消費しないようカリング（破棄）します。
  3. 画面に少しでも触れている場合は、座標を 16bit 整数の限界 (`INT16_MIN` ～ `INT16_MAX`) の範囲内に安全に丸め（サニタイズ）てから、コアAPIへ `int16_t` として渡します。

これにより、Cエンジンは不正な値から完全に守られつつ、コア関数は本来のシンプルな形を保つことができました。

## Python API のユーザビリティ向上: Float の自動変換

ゲームコードを書く上での大きな不満点も同時に解消しました。

- **課題**: これまでバインディング関数は `mp_obj_get_int()` を使用していたため、物理演算等で生じた `float`（浮動小数点数）を渡すと `TypeError` となっていました。そのため、ゲーム側で毎フレーム `int(x)` と明示的にキャストする必要がありました。
- **解決策**: バインディング層に `get_int_from_obj()` という補助関数を追加しました。これにより、引数が `float` の場合は自動的に `int` へ変換（切り捨て）されるようになり、Python側のコードがよりスッキリ書けるようになりました。

## ファームウェアビルドの修正

- `modlightengine.c` 内に、以前のリファクタリングで存在しなくなっていた `#include "sound_engine/sound_engine.h"` が残存しており、ESP32 (Cardputer) 環境でのファームウェアビルドエラーを引き起こしていました。
- **補足**: `sound_engine` は現在、MicroPython側へ独立したモジュール (`_sound_engine`) として登録されており、C言語空間のグラフィックエンジン (`modlightengine.c`) から直接呼び出されることはありません。そのため、ヘッダのインクルード自体が不要でした。これを削除することで、クロスプラットフォームでの完全な再ビルドに成功しました。
