# Walkthrough: バイナリフォーマット (`.ubgm`) と 新MMLフォーマット (`.mml`) の導入

ESP32などのハードウェア制約の強い環境で発生していた、巨大なMML文字列のパースによる `MemoryError` を解決するため、サウンドエンジンに専用のバイナリフォーマット (`.ubgm`) と、その元となる可読性の高いテキストフォーマット (`.mml`) を導入しました。

## 実装内容
1. **新MMLテキストフォーマット (`.mml`) の定義**:
   - 従来のカンマ区切りの文字列形式では長大な曲の可読性が下がるため、小節単位等のブロックで複数チャンネルの音符を視覚的に並べられるフォーマットを新たに導入しました。
   - `0:` のようにチャンネル番号を指定して記述でき、同一チャンネル内での自由な改行もサポートしているため、タイ（`&`）が連続する長いフレーズでも横スクロールせずに綺麗に整形できます。
2. **`.ubgm` 形式の仕様策定**: 
   - 1音符をわずか6バイト（周波数2, 長さ2, ボリューム1, 波形タイプ1）に圧縮。
   - ヘッダにメタデータ（トラック数、オフセット等）を持たせ、PCでもESP32でも即座に読み出せる設計にしました。
3. **コンパイラの追加 (`tools/mml2ubgm.py`)**: 
   - `.mml` テキストファイルをパースし、`.ubgm` バイナリファイルに変換するコマンドラインツールを追加しました。
4. **エンジンの対応 (`engine/sound.py`)**: 
   - `load_ubgm(filepath)` および `play_ubgm(filepath)` を追加しました。
   - `_UBGMTrack` という「遅延評価リスト」クラスを実装し、HAL側の配列アクセスコードを**一切変更せず**にシームレスにバイナリデータを再生させることに成功しました。
5. **テストアセットの移行**:
   - `upy_light_engine/main.py` にハードコードされていたMML文字列を `assets/sounds/main_chord.mml` と `main_loop.mml` に抽出し、`.ubgm` 形式で再生するように書き換えました。

## 検証
- テストスクリプト (`test_ubgm.py`) を作成し、`.ubgm` のコンパイルおよび再生が CPython 上でエラーなく正常に動作することを確認しました。

> [!TIP]
> 既存の `engine.sound.play_mml` も引き続き利用できるため、後方互換性は完全に保たれています。次はゲーム側（`motorcycle_cardputer`）のアセットをこの新フォーマットに移行する準備が整いました！
