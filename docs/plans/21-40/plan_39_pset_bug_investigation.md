# Plan 39: pset におけるクラッシュ可能性の調査計画

## 概要

`TODO.md` に記載されている「psetでのクラッシュの可能性を調査し、必要であれば修正する」という課題に対応するため、調査のみを目的とした計画を立案します。
現時点では具体的なクラッシュの再現手順や条件が不明であるため、様々な境界値や異常値を意図的に注入し、CエンジンレイヤーおよびPythonバインディング層での堅牢性を検証します。

## 調査の背景と仮説

`pset` (Pixel Set) やそれに依存する描画処理 (`line`, `fill_rect` 等の直接バッファアクセス) は、フレームバッファのメモリ領域に直接書き込むため、バウンダリチェック（範囲外アクセス防止）が漏れると C/C++ レベルのセグメンテーションフォールトや MicroPython でのフリーズ（Hard Fault）を引き起こします。

現在の Cエンジン (`c_modules/core/engine_render.c`) には以下のようなバウンダリチェックが存在します。
```c
if (x >= 0 && x < framebuffer->width && y >= 0 && y < framebuffer->height) {
    framebuffer->buffer[y * framebuffer->width + x] = (uint8_t)color;
}
```
一見安全に見えますが、以下の要因でクラッシュを引き起こす可能性（仮説）が考えられます。

1. **整数オーバーフロー / アンダーフロー**:
   引数 `x`, `y` は `int16_t` (C側) です。極端な負の値（例: `-32768`）や、`render_line` 等のアルゴリズム内部の計算（`dx, dy, err`等）でオーバーフローが発生して予期せぬメモリアクセスが発生していないか。
2. **型の不一致とキャストエラー**:
   Python 側で `float`, `None`, または巨大な整数 (`> 32767`) が渡された際、MicroPython の Cモジュールバインディング層で正しく例外が投げられずにメモリ破壊を起こしていないか。
3. **色 (color) の異常値**:
   カラーパレット外の値（例: `-1`, `0xFFFFFFFF`）を渡した際のキャストの安全性。

## 調査手順

### 1. 限界テストスクリプトの作成
プロジェクト内に一時的な検証用スクリプト（例: `tests/tmp/test_pset_crash.py`）を作成し、`pset`, `line`, `rect` 等に対して以下のテストケースを自動実行するコードを実装します。

* **座標の境界値**: `(-1, -1)`, `(240, 135)`, `(INT16_MAX, INT16_MAX)`, `(INT16_MIN, INT16_MIN)`
* **巨大な数値**: `(99999, 99999)` （16bitに収まらない値）
* **不正な型**: `float` (`10.5`), `None`, `str` (`"10"`)
* **不正な色**: `-1`, `256`, `0xFFFFFF`

### 2. マルチ環境での実行・検証
作成したテストスクリプトを以下の各環境で実行し、挙動（安全に例外が出るか、それとも完全にクラッシュ・フリーズするか）を観察します。

1. **CPython 環境 (Windows/PC)**:
   ネイティブDLL (`core_engine_win.dll`) を利用して実行。セグメンテーションフォールト等が発生しないか確認。
2. **MicroPython 環境 (Cardputer 実機)**:
   実機に転送して実行。シリアルコンソール経由で `Hard Fault` や無言フリーズが発生しないか確認。
3. **WASM 環境 (ブラウザ)**:
   ブラウザの開発者コンソールで `RuntimeError: memory access out of bounds` 等の WebAssembly 固有エラーが出ないか確認。

### 3. Cコードの静的解析（机上チェック）
`engine_render.c` の `render_line` や `render_draw_sprite` 等、間接的にピクセル操作を行うコード内のバウンダリチェックや整数型の扱いに穴がないか、ソースコードレベルで精査します。

## 期待される成果

* **クラッシュの再現手順の確立**: もしバグが存在すれば、どの値・どの関数で落ちるのかが特定できる。
* **原因の特定**: C言語側の問題か、MicroPythonラッパー側の問題かが切り分けられる。
* **次回への対応方針**: 調査結果をもとに、修正方針（バリデーションの追加、型の修正など）を決定し、次の実装フェーズへ繋げる。
