# Phase 3c: 非同期 SPI DMA & Core 1 レンダリング実装

## 概要

このフェーズでは、アーキテクチャの完全な非同期ハードウェアレンダリングへの移行を完了しました。ディスプレイリストの描画（ピクセル化）と、SPIによるDMA転送の両方を、ESP32-S3の「Core 1」にFreeRTOSタスクとしてオフロードしました。

これにより、Python側（Core 0）が画面の描画やSPI転送の完了を同期的に待たなければならなかった大きなパフォーマンスのボトルネックが解消され、グラフィックスハードウェアの処理完了をPythonが待機する時間がわずか **0.16 ms** にまで短縮されました。

## 主な変更点

1. **FreeRTOS レンダリングタスク (`render_task`)**
   - C言語モジュール (`modlightengine.c`) 内に、Core 1に固定（Pinned）された専用のFreeRTOSタスクを作成しました。
   - このタスクは、FreeRTOSのキュー (`render_queue`) を介して送られてくる `RenderJob` 構造体を待ち受けます。
   - ジョブを受け取ると、Core 1は直ちにC言語関数の `render_display_list()` を実行してピクセルを描画し、その後すぐにSPI DMA転送を開始します。

2. **チャンク分割DMA (Chunked DMA) とダブルバッファリング**
   - 当初、240x135ピクセル全画面分のフレームバッファを2つDMA対応メモリ（SRAM）に確保しようとしましたが、ヒープの断片化により `MemoryError: Failed to allocate DMA buffer` が発生しました。
   - そこで、画面を横にスライスして15ラインごとの「チャンク」に分割する手法 (`CHUNK_HEIGHT = 15`) を導入しました。
   - Core 1は最初の15ラインを `dma_chunk_bufs[0]` にレンダリングし、`spi_device_queue_trans()` でDMA転送のキューに入れた後、**DMA転送が行われている裏で直ちに次の15ラインを `dma_chunk_bufs[1]` にレンダリング開始**します。
   - この工夫により、必要なDMAメモリサイズが約128KBからわずか **14.4KB (7.2KBのバッファ2個分)** に劇的に削減され、メモリ不足を完全に回避しつつ最大スループットを維持できるようになりました。

3. **Python API のアップデート**
   - `_lightengine.submit_and_send()`: Python側はディスプレイリスト（描画コマンドのリスト）を構築した後、この関数を呼ぶことで、処理をブロックすることなく「ディスプレイリスト」と「パレット」をCore 1に丸投げします。
   - `_lightengine.sync()`: *次の* フレームを描画依頼する直前に呼ばれ、Core 1の処理が確実に終わっていることを保証します。
   - **バグ修正**: 当初、ディスプレイへの描画領域を指定する `set_window` コマンドが送信されておらず、画面がノイズ（砂嵐）になる現象が発生しましたが、Pythonのループ内で `submit_and_send()` の直前に `set_window` を送るように修正し、解決しました。

## プロファイリング結果とパフォーマンス

`main.py` でプロファイラを有効にした状態での計測結果は以下の通りです：

```text
[PROFILE] update: ~5.5 ms
[PROFILE] draw_all: ~19.5 ms
[PROFILE] sync: 0.16 ms
```

### 分析:
- **`sync: 0.16 ms`**: これが最も重要な指標です！Pythonが次のフレームのための `update()` と `draw_all()` の計算（合計約25 ms）を行っている裏で、**Core 1 は完全に前のフレームのレンダリングとSPI転送を終えていた**ことを意味します。もはやPythonは画面のハードウェア処理を待つためにブロックされることはありません！
- **ハードウェアの理論上の限界値**: グラフィックスエンジン自体（Core 1 + SPI）は、60 FPSの制限時間である16.6ms以内に完全に作業を終えられる驚異的なパフォーマンスを叩き出しています。
- **現在のボトルネック**: Pythonの `draw()` 関数（特にテキスト描画やアニメーションスプライトのための `math.sin` などの計算）に約 19.5 ms かかっています。このため、$1000 / (19.5 + 5.5) \approx 40 \text{ FPS}$ となり、描画するテキストやスプライトの量によって40〜50 FPSの間で動作しています。

ハードウェアエンジンはこれ以上ないほどに最適化され、完全な非同期動作を実現しました。これ以上のFPSの向上は、Pythonのゲームロジックそのものの最適化（描画するテキストを減らす、数学計算の最適化、ループ処理への `Viper` や `Native` エミッタの導入など）によってのみ達成されるフェーズに入りました。
