## 2026-06-14: HALクラス構造の分離と `blt` の汎用化

**【指摘内容】**
> 元々はスプライト(ARGB4444)とFramebuffer(RGB565)は別のクラスであると想定してたけど、同じクラスとしてパフォーマンス上問題なければそれでもいいです。その場合はフォーマットを外部から識別できるようプロパティなどを用意して、またbltの引数がどちらでも動くようにした方がいいと思います。microPythonの場合で実装する場合のことも考慮してパフォーマンスについては検討して下さい。
> 
> *(HALを使う側からは「Framebufer」という名前でアクセスできるといいと思います。)*

**【設計判断と対応】**
MicroPython環境におけるパフォーマンスとメモリ効率を考慮し、クラスを完全に分離する設計を採用しました。
- **`Image` クラスの新設**: ARGB4444フォーマットのスプライトデータを保持する専用クラス。MicroPython環境では `framebuf.FrameBuffer` を継承しないただのデータコンテナとすることで、初期化のオーバーヘッドとメモリ消費を最小限に抑止。
- **`Framebuffer` への名称変更**: `PyxelFrameBuffer` から名称を変更。RGB565の画面バッファを管理し、MicroPython環境では標準の `framebuf.FrameBuffer` を継承してネイティブの描画API(`rect`, `fill`等)を利用可能に。
- **`blt` の自動判定化**: 引数に渡されたオブジェクトの `.format` を確認し、`Image` (ARGB4444) であればアルファブレンド合成を、`Framebuffer` (RGB565) であれば `colkey` を使った透過コピーを自動的に行うように修正。

## 2026-06-14: アルファブレンド演算の高速化（ビットシフト化）

**【指摘内容】**
> aは0,15の場合はもう考慮しなくていいから、いっそのこと inv_a = 16 - a として // 16 を >> 4 としたら速くなることってない? もしそうなら検討してくれる? 速くならないのなら現状のままでいいよ。
> 
> *(対象コード: `out_r = (sr * a + dr * inv_a) // 15`)*

**【設計判断と対応】**
非常に有効な最適化テクニックであると判断し、実装に取り入れました。
- 完全に透明(`a=0`)と完全に不透明(`a=15`)のケースは事前に `continue` 等で分岐処理されているため、実際のブレンド処理では `a` は 1〜14 の値をとる。
- したがって、`15` での除算を `16` での除算で近似（`inv_a = 16 - a` とし、`>> 4` のビットシフトを使用）しても、各色成分の最大値スケールは崩れずに正しい結果が得られる。
- **効果**: マイコン環境（MicroPython + Viper）において、比較的重い「除算命令」を排除し、1サイクルの「ビットシフト命令」に置き換えることができたため、描画ループの大幅な高速化が実現した。

## 2026-06-21: Cardputer ADV キーボード (TCA8418) の対応と物理キー・論理ボタンの分離

**【指摘・要望内容】**
> - Cardputer ADVに搭載されているTCA8418マトリクスキーボードから入力を取れるようにしたい。
> - `Key_Up`, `Key_Down` のような物理キーと、`Button_Up`, `Button_A` のようなゲームの論理ボタンを分けて定義したい。プレフィックスは `KEY_` ではなく `Key_`、`Button_` が好みで、定数は `enum.auto()` で綺麗に定義したい。
> - PC環境ではWASDとN, Mキー、Cardputer環境ではWASDとSpace, Enterキーという風に環境ごとの初期マッピングを設定したい。
> - 抽象層（`input.py`）ではなく、HAL層に環境ごとのマッピング処理を閉じ込めたい。

**【設計判断と対応】**
- **TCA8418 マトリクススキャナ実装 (`tools/test_tca8418.py`, `hal/input_micropython.py`)**:
  実機用のテストスクリプトを作成して物理配線を調査し、単純な連番配線ではないこと（Space=68, Enter=67 等）を解明。I2C通信で取得した生のマトリクスコードから物理キー定数へ変換するマッピングテーブルをHAL内に実装した。
- **定数のリファクタリング (`constants.py`)**:
  物理キー定数 (`Key_W`, `Key_Space`等) と論理ボタン定数 (`Button_Up`, `Button_A`等) を分離し、`enum.IntEnum` と `auto()` を用いてクリーンに再定義した。標準で `enum` モジュールが存在しない MicroPython 環境でも動作するよう、`try-except` によるフォールバック処理を追加した。
- **抽象層とHALの責務分離 (`input.py`, `hal/input_*.py`)**:
  `input.py` にあったマッピング処理を削除し、各プラットフォーム固有の `_button_to_key` マッピングを `hal/input_cpython.py` および `hal/input_micropython.py` の内部に移動させることで、プラットフォーム依存コードをHAL内に完全に隠蔽する設計とした。

* **TCA8418キーマッピングの実装とハードウェア抽象化 (2026-06-20)**
  * **指摘・要望**: PC環境とCardputer環境で同じ `main.py` が動くようにしたいが、`input.py` 内に直接TCA8418のキーコードをハードコードするのはプラットフォーム依存になるため美しくない。
  * **対応**: `constants.py` を作成し、定数を `Key_` (物理キー) と `Button_` (論理ボタン) に分離。`input.py` は単なるルーティング層とし、プラットフォーム固有のキーマッピングは `hal/input_cpython.py` と `hal/input_micropython.py` にカプセル化した。

* **ログ出力制御（ロギングシステム）の実装 (2026-06-21 09:30)**
  * **指摘・要望**: コンソールログは便利だが、本番時や不要な時に一括で出力をオン・オフできる仕組みが欲しい。
  * **対応**: MicroPythonのメモリ制約を考慮し、超軽量な `logger.py` モジュールをルートディレクトリに新設。`main.py` や `hal/framebuffer_micropython.py` 内の `print` を `logger.debug` に置き換え、`logger.DEBUG_ENABLED = True / False` だけで全てのデバッグログを制御できるようにした。

## 2026-07-04: PCエミュレータにおける60FPS化への最適化と、Cエンジンの固定小数点演算の確認

**【指摘・要望内容】**
> main.py には `fps=60` が指定されているが、PC環境で40〜45FPSしか出ない。Cardputer Adv実機での60FPS達成が困難そうであれば、今のうちにPCでボトルネックを潰しておきたい。

**【設計判断と対応】**
- **PCエミュレータ側のタイマー計算バグ修正 (`engine/hal/framebuffer_cpython.py`)**:
  Tkinterの `_root.after()` に渡す待機時間を固定で `1000 // fps` (16ms) としていたため、「1フレームの処理時間＋16ms」となり意図せずFPSが低下していました。`time.time()` を使って実際の処理時間を計測し、動的に待機時間 (`delay`) を算出する方式に修正しました。
- **PILのネイティブパレット機能への移行 (`engine/hal/framebuffer_cpython.py`)**:
  Cエンジンが描画した INDEX8 形式のバッファを、PC上で表示するために Python の `for` ループで1ピクセルずつ RGB888 に変換していた処理が巨大なボトルネック（約10〜20ms）となっていました。PIL (`Image.frombytes("P", ...)`) のネイティブパレット機能を直接呼び出すことで、C言語レベルでの高速変換を実現し、PCエミュレータでも安定して 60FPS が出るように最適化しました。
- **Cエンジンの内部最適化状態の確認 (`engine_render.c`)**:
  Cエンジン内で `render_draw_sprite` 等の描画ループに浮動小数点数 (`float`) が混入しているのではないかという懸念がありましたが、実装を確認したところ、スケール・回転処理はすでに高速な固定小数点数 (`sin_inv_fp`, `cos_inv_fp` 等) による整数演算のみで実装されていました。Cardputer Adv (ESP32-S3) では、Cエンジンの描画自体は 1〜2ms で完了し、SPI DMA 転送 (1.6ms) を合わせても 60FPS (16.6ms) の猶予内に十分収まる見込みであるため、Cコアにこれ以上の過度な最適化は現時点では不要と結論づけました。

## 2026-07-04〜07-05: Phase 3 (MicroPython組み込みとハードウェア連携) のレビュー

**関連計画**: `plan_30_c_cpp_engine_architecture.md` (Phase 3b〜3c)

### 1. 音楽再生スレッドのGIL Starvation問題
* **指摘事項/現象**: Cエンジン側が60FPSで動作し、サンプル `main.py` が45FPSで動作している状態で、バックグラウンドの音楽再生が本来の10倍ほど遅く（間延びして）再生される。
* **原因分析**: `_thread` を用いて音楽シーケンサーを走らせていたが、メインループ（描画ロジック）が重く `time.sleep_ms(0)` での待機余力がなくなった結果、MicroPythonのGIL（グローバルインタプリタロック）がメインスレッドに占有され、音楽スレッドに処理が回らなくなっていた。
* **対応方針**: 音楽シーケンサーの `_thread` を廃止し、ゲームのメインループ `update()` の中で `sound.update()` を定期的に呼び出して音符を進めるポーリング（ステートマシン）方式に設計を変更。遅延は完全に解消した。

### 2. 初期化ブロックによる音楽再生タイミングのズレ
* **指摘事項/現象**: MML再生時、最初の2音だけがゆっくりと再生され、その後は正常に戻る現象が発生。
* **原因分析**: `fb.run()` の直前で `play_mml()` を呼び出していたため、最初の音符が鳴り始めた直後に、`fb.run()` 内部での重い初期化処理（ディスプレイリストのアロケート等）が走り、次の音へ進むための `sound.update()` が呼ばれるまでに遅延が生じていた。
* **対応方針**: 音楽の再生開始を、初期化完了後のメインループ内（例: `frames == 120` 到達時）に移動。また、ユーザー要望により MML パーサーに `V` (ボリューム) コマンドを新規実装し、音量を絞って再生できるようにした。

### 3. パフォーマンスとアーキテクチャへの懸念 (Pythonの速度問題)
* **指摘事項/現象**: 「単純な `main.py` で45FPSしか出ないなら、本番のゲーム（motocycle）を30FPSで動かすのは絶望的ではないか？ 最初からすべてC言語で移植すべきでは？」という根源的なアーキテクチャへの懸念が提起された。
* **原因分析**: プロファイリングの結果、以下の事実が判明した。
  1. `draw_text` に約4.5msかかっている。これはテキストを1文字ずつPythonループ内で計算し、個別にSprite描画APIを叩くという、Pythonが最も苦手とする処理を行っていたため。
  2. `draw_rects` のループ内で毎フレーム `math.sin` や浮動小数点演算を行っており、これに2〜4ms消費していた。
* **対応方針**: 
  * C言語へ全体を移植するのではなく、**「描画や音響はC、ゲームロジックはPython」という現在のアーキテクチャを堅持**する。
  * 明らかなボトルネックである「Pythonでのミクロな文字描画処理」をCモジュール側へ移植する (`push_draw_text`) ことを `TODO.md` に追加した。
  * 実際のゲームロジックでは、浮動小数点演算を避け、整数計算やテーブル参照、あるいは `@micropython.viper` を活用することでPythonのままでも十分な速度が出ると結論づけた。

## 2026-07-05: サウンドエンジン（I2S）の実装と波形・和音への対応

**関連計画**: `plan_34_sound_engine.md` / `plan_35_pc_software_synth.md`

**【開発の背景・要件】**
- `Cardputer` に内蔵されているスピーカ（I2Sオーディオ, ES8311チップ）から音を鳴らす仕組みの構築が必要になった。
- MML記法による単音BGMの再生（`play_mml`）を実装していたが、よりリッチなゲームサウンド体験を実現するため、以下のような要件が追加された：
  - 複数トラック（和音）の同時再生。
  - レトロゲーム風の波形（矩形波、ノコギリ波、三角波、ノイズ波）の切り替え機能。
  - リニアディケイ（音が徐々に減衰するエンベロープ）の適用。
  - OSネイティブのPCシミュレータ上でも、外部ライブラリ(`PyAudio`等)を追加せずに全く同じ音質・仕様で鳴らせること。

**【設計判断と対応】**

### 1. Cardputer実機向けのCモジュール化 (`sound_engine.c`)
- **ハードウェアミキシング**: 4チャンネルのミキシングをC言語側で計算。ESP32のFreeRTOSタスク (`sound_task`) を使って非同期にオーディオバッファ（I2S）へ書き込むアーキテクチャを採用した。
- **波形生成**: 矩形波、ノコギリ波、三角波の計算式を実装。ノイズ波については、計算機科学の定番である "Numerical Recipes in C" 由来の線形合同法 (LCG) の定数（`1664525`, `1013904223`）を採用して軽量かつ良質なホワイトノイズを生成するようにした。
- **リニアディケイ**: 音の切り替わりでの「プツッ」というノイズを防ぎ、レトロ風のパーカッシブな音を作るため、0.5秒かけて音量が0になる線形減衰エンベロープを実装した。
- **課題と解決**: `sound_engine.c` によるCモジュールをロードすると、MicroPythonのメインループとの兼ね合いで「USBシリアル通信が過負荷で切断される」問題が発生した。これは `i2s_channel_write` がブロックされずに即座に戻ってしまい、`while(1)` ループが過剰にCPUを占有していたためと判明。バッファ書き込み時や無音時に `vTaskDelay` や適切なブロッキング時間 (`portMAX_DELAY`) を挟むことで回避した。

### 2. PC版 ソフトウェアシンセサイザーの自作 (`sound_synth.py`)
- **PyAudioの回避**: ユーザーのPC環境に依存せず、`pip install` 等の手間を省くため、Python標準ライブラリのみでWAVのバイト列を動的生成する「ピュアPythonシンセサイザー」を開発した。
- **実機の完全エミュレーション**: C言語エンジンと全く同じ波形計算、エンベロープ、LCGノイズのアルゴリズムを移植し、全く同じ音質で出力できるようにした。
- **パフォーマンス最適化**: 生成したオーディオサンプルを格納・結合する際、巨大なリストに対する `for` ループでの `struct.pack` 呼び出しは非常に遅いことが判明。これを `array.array('h')`（型付き配列）に書き込み、一括で `.tobytes()` 変換する最適化により、生成時間を数十分の1（0.05秒以下）に短縮した。
- **Mac / Windowsの統合**: PC版のHALとして存在していた `sound_cpython_win.py` と `sound_cpython_mac.py` は、WAV生成ロジックの自作により「OSネイティブの再生API（`winsound` / `afplay`）を叩く部分」以外は完全に重複していたため、`sound_cpython.py` として1つのファイルに統合（DRY化）し保守性を高めた。

### 3. API設計
- `mml_parser.py` を改修し、MML文字列をカンマ `,` で区切ることで複数トラックを同時に定義できるようにした（例: `T100 O4 C4, T100 O4 E4, T100 O4 G4` でCメジャー和音）。
- 効果音（ジャンプ音、コイン音など）を簡単に鳴らすための `sound.play_sfx(name)` APIを公開し、`main.py` 側でボタンの入力エッジ検出（押した瞬間だけ判定）と組み合わせて呼び出せるようにした。
